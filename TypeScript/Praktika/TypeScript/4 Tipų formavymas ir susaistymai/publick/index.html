Assertations - prielaidos

Nes jeigu mes turime kintamąjį, kuris nėra užpildytas deuomenims tuo momentu kode, o turi būti kažkur iš kažkur paimta ir priskiriama - TS'as įžiūri atvejį, kai tipas gali būti null ar undefined, o ne tai ko galbūt programuotojas tikisi. Types VS Interface

Interface gali būti naudojamas kuriant klasių arba objektų aprašymą.
Type gali būti naudojamas kuriant betkokį aprašymą.

Interface gali būti pratęstas kitų Interface naudojant 'extends' prierašą.
Type gali būti pratęstas kitu Type naudojant '&' (interaction) ženkliuku.

Interface gali būti papildytas naujomis savybėmis/metodais, negrįžtant prie originalaus Interface aprašymo, o tiesiog kreipiantis į Interface neegzistuojantį tipą ir jį nurodant. (main.ts - line 22)
Type gali būti papildytas naujomis savybėmis/metodais/reikšmėmis/..., bet norint tai atlikti būtina sugrįžti prie tos vietos, kur jis originaliai skurtas.

Kuriant Interface negalima nurodyti vieno arba kito varianto. Reikia susikurti du Interface, su vienu ir kitu variantu ir kuriant duomenį naudoti vieną arba kitą Interface
Kuriant Type galima nurodyti vieną arba kitą variantĄ

Išvada (mano asmeninė (Rokas Banaitis)): Siūlyčiau naudotūi Type vietoj Interface. Nes Type gali atlikti beveik viską ka atlieka Interface, o taip pat daug kitų dalukų (lanksčiau), kurių Interface negali.

Neprivalomos tipų savybės

Kuriant objektų tipus, galime nurodyti, kad tam tikros savybės nėra būtinos. Tai labai lengvai nurodoma prirašant "?" ženklą prie savybės pavadinimo.
